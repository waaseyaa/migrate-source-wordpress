## [Unreleased]

### Fixed

- **`WpCommentsToEngagement` id-map lookup type mismatch**: `WordPressCommentSource::normalize()` emits `post_id` as `int` and `parent_id` as `int|null`, but every `SourceId` in this package (including `WordPressPostSource::sourceIdFor()` and `WordPressCommentSource::sourceIdFor()` itself) keys its id-map row with a string id. `SourceId` hashing is type-sensitive, so feeding the raw int straight into `LookupProcessor` hash-missed on every single comment record — `post_id` and `parent_id` silently resolved to `null` on every migrated comment, with no error surfaced. Flagged by an adversarial verifier reviewing the comments migration (which has never yet been run against a production destination — this closes the gap before it is). Fixed by chaining a `TypeCoerceProcessor('string')` ahead of both `LookupProcessor` calls, matching the established pattern already used by `WpPostsToArticles`'s `parent_ref` and `WpPostsToPathAliases`'s post lookup; `null` parent ids continue to pass through unchanged (top-level comments correctly resolve to `null`, not a miss).

## [0.1.0-alpha.2] - 2026-07-10

### Added

- **`WordPressPostSource` trash-skip default (G-021)**: records with `wp:status` === `trash` are now skipped by default. Pass `includeTrashed: true` to the constructor to opt back in to importing them.
- **`WordPressPostSource` status contract pinned (G-021)**: the `status` field was already emitted as the raw WordPress status string (`publish`, `draft`, `pending`, `private`, `future`, `trash`, ...) — this is now covered by a regression test and documented as a stable contract. Consumers that need a boolean "published" flag must add an explicit status→bool mapping step in their own process map; this source will never flatten status itself.
- **`WordPressPostSource` post_type allowlist filter (G-027)**: constructor gains `?array $postTypes = null`. When provided (a non-empty list of non-empty `post_type` strings), only matching records are emitted — enabling a per-bundle migration split (e.g. separate `WordPressPostSource` instances for `page`, `post`, and a custom `event` type feeding three distinct migrations). `null` (default) preserves the existing behavior of emitting every post type.
- **Postmeta extraction process plugin** (`Process\WordPressPostmetaExtract`, id `wordpress_postmeta_extract`) — pulls one named key out of a post's WXR `_extra['postmeta']` map into a typed destination field, tolerating missing keys and malformed shapes (G-016).
- **Events Calendar recipe**: three new default `MigrationDefinition` factories in `Migration\` covering WordPress's The Events Calendar plugin shape — `WpEventsToNodes` (`tribe_events`, extracts `_EventStartDate`/`_EventEndDate`/`_EventOrganizerID`/`_EventVenueID`), `WpOrganizersToNodes` (`tribe_organizer`, extracts `_OrganizerEmail`/`_OrganizerPhone`/`_OrganizerWebsite`), `WpVenuesToNodes` (`tribe_venue`, extracts `_VenueAddress`/`_VenueCity`/`_VenueProvince`/`_VenueCountry`) (G-016).
- **Docs**: "Events (The Events Calendar)" recipe in `docs/migrating-from-wordpress.md`; "Mapping arbitrary postmeta" section in `docs/customization.md` documenting the `WordPressPostmetaExtract` chain pattern (G-016).
- **Menu migration** (G-022): `Source\WordPressMenuSource` (`SOURCE_TYPE = 'wp_menu_item'`) filters WXR `nav_menu_item` posts into a flat menu-item shape — `title` (falls back to a caller-supplied `objectTitles` WP-object-id→title index, built via the new `WordPressMenuSource::objectTitleIndex(WxrReader): array<int, string>` static, when the item's own `<title>` is empty — WordPress leaves it empty for `post_type`/`taxonomy`-flavour items; verified against the real SFN export where 15 of 17 `nav_menu_item` records ship with an empty title), `url` (custom links), `object_type`/`object_id` (post/term links, for app-side id-map lookup), `menu_name` (the item's `nav_menu` term slug), `parent_wp_id` (raw WordPress parent id, `'0'` normalized to `null`), `weight` (`wp:menu_order`), `enabled`. `Migration\WpMenusToMenuLinks` (`MIGRATION_ID = 'wp_menus_to_menu_links'`) is the default factory targeting `waaseyaa/menu`'s `menu_link` entity type; see `docs/migrating-from-wordpress.md` "Menus" for the wiring recipe (including the title-fallback index wiring) and its parent/URL-resolution caveats.
- **Elementor page-builder decoder** (`Process\WordPressBuilderContentDecode`, G-013): Elementor-built pages store their real content as a JSON tree in the `_elementor_data` postmeta entry, not in `post_content` — the published distribution previously had no decoder for that JSON, so Elementor-built pages imported with an empty body while their content sat unmigrated (observed on real pass-1 data: the SFN home page imported with body length 0 against 52,479 chars of `_elementor_data`). The new plugin walks the tree (sections/containers > columns > widgets) via an internal `PageBuilder\ElementorTreeDecoder`, produces a normalized block tree that keeps layout intent (bands, cards, grids) and design intent (background variant, min-height, alignment, icon/button intent), and renders it to clean semantic HTML through `PageBuilder\BlockRenderer` + `PageBuilder\HtmlCleaner` — no `elementor-*` classes, inline styles, or scripts survive. Ported from the framework's former `waaseyaa/page-builder` package's `ElementorDecoder` decoding strategy, adapted into the connector's process-plugin shape (no decoder registry/service-provider). Decoding only runs when postmeta `_elementor_edit_mode === 'builder'` — WordPress itself only renders `_elementor_data` under that condition, and `_elementor_data` can remain present as stale residue from a past edit-mode toggle while `post_content` is the actual live body (verified against the real SFN export: post 975 "1850 Robinson Huron Treaty" carries `_elementor_data` with no `_elementor_edit_mode` key, and decoding it unconditionally would have replaced 6,126 chars of live content with 724 chars of stale output). Outside that gate, or when `_elementor_data` is absent/empty (`''`/`'[]'`), the content field passes through unchanged; when the gate passes but the payload is malformed JSON or decodes empty, the original content is kept and a warning is logged via the plugin's `Psr\Log\LoggerInterface` seam.
- **Gutenberg block-comment stripping** (same plugin, G-029): `WordPressShortcodeStrip` never touched Gutenberg's `<!-- wp:name ... -->` / `<!-- /wp:name -->` block-delimiter comments, so migrated Gutenberg-authored bodies retained the literal markers. `WordPressBuilderContentDecode` now strips them unconditionally from its output (both the Elementor-decoded HTML and any pass-through content), preserving the inner HTML.
- **Real-payload test fixtures** under `testing/Fixtures/` extracted from the Sheguiandah First Nation pass-1 WXR export: `elementor-sfn-home.json` (the 52,479-char home page payload), `elementor-sfn-footer.json` (a smaller real payload), and `gutenberg-sfn-election-notice.html` (a real Gutenberg-authored post body).
- **Path-alias emission + stock media URL resolver (G-020)**: hierarchical WordPress URLs (e.g. `/members/rht/`) previously had no destination-side equivalent and `WordPressMediaRewriteUrl` had no default resolver implementation, so a real migration left 5 flat-slug collisions and 97 in-body `/wp-content/uploads/` references unrewritten. Closed by:
  - `Wxr\WxrReader` now captures the WXR `<link>` permalink under each post/attachment's `_extra['link']` — additive, does not change the existing typed record shape.
  - `Process\WordPressPermalinkToAlias` (id `wordpress_permalink_to_alias`) normalizes a permalink into a destination alias: domain and query string stripped, trailing slash stripped, percent-decoded for non-ASCII slugs. Returns `null` for querystring-only ("plain") permalinks and the bare homepage link — see its docblock for the full contract.
  - `Process\UuidToSystemPathProcessor` (id `wordpress_uuid_to_system_path`) bridges a `LookupProcessor`-resolved destination uuid to a system path (e.g. `/node/42`) via an app-supplied `\Closure(string $entityType, string $uuid): int|string|null`.
  - `Migration\WpPostsToPathAliases` (`MIGRATION_ID = 'wp_posts_to_path_aliases'`) — default factory targeting `waaseyaa/path`'s `path_alias` entity type, composing the above with a `TypeCoerceProcessor` + `LookupProcessor` chain against the posts migration's id-map. Depends on the posts migration. Constructor gains `?SourcePluginInterface $source = null` (mirroring `WpEventsToNodes`'s seam): left unset, the migration defaults to an unfiltered `WordPressPostSource`, which emits an alias attempt for every WordPress post type — including non-content records like `nav_menu_item`, `custom_css`, `wp_global_styles`, and `wp_navigation` that the posts migration itself never wrote. Pass the SAME filtered source (or the same `postTypes` allowlist) you used for your posts migration.
  - `Media\IdMapMediaUrlResolver` — stock `\Closure(string $relativePath): ?string` resolver for `WordPressMediaRewriteUrl`, built from an uploads-relative-path→attachment-id index (`indexFromSource()`, built straight from `WordPressMediaSource`) plus the media migration's id-map plus an app-supplied uuid→URL closure. Any chain miss returns `null` and logs a warning (leave-untouched semantics).
  - **Docs**: "URL preservation (path aliases + media URL rewriting)" section in `docs/migrating-from-wordpress.md` covering wiring, the alias normalization contract, the source-seam requirement, and the required run order (path aliases after posts; media rewriting needs the media id-map populated).
- **Id-map reference resolution (G-019)**: opt-in wiring — via the new `Migration\ReferenceResolutionOptions` DTO passed as `$references` to `WpPostsToArticles`, `WpTermsToTaxonomy`, and `WpMediaToEntities` — that resolves cross-migration references through the real `migration_id_map` instead of leaving them as raw WordPress ids/slugs/logins. Every resolved value lands in a NEW destination field; the previously-shipped raw fields (`author_login`, `parent_id`, `terms`, `parent_slug`, `parent_post_id`) are emitted unconditionally, unchanged — existing consumers that never pass `$references` see byte-identical process maps.
  - New process plugins: `Process\WordPressEntityRefResolve` (`wordpress_entity_ref_resolve`) converts a `LookupProcessor`-produced destination UUID into a storage identifier via an application-supplied `(string $entityType, string $uuid): int|string|null` closure — the seam this connector needs because `LookupProcessor` only ever returns UUID strings, while many destination reference fields (`Node.uid`, `Term.parent_id`) are integer foreign keys. `Process\WordPressAuthorIdResolve` (`wordpress_author_id_resolve`) bridges a post's `author_login` (WXR `dc:creator`) into the numeric `wp:author_id` `WordPressUserSource`'s id-map is actually keyed by. `Process\WordPressTermsResolve` (`wordpress_terms_resolve`) and `Process\WordPressTermParentResolve` (`wordpress_term_parent_resolve`) resolve a post's `terms` list / a term's `parent_slug` via a `"{taxonomy}:{slug}" → wp:term_id` index (slugs are only unique within a taxonomy).
  - New static index builders: `WordPressUserSource::loginIndex(WxrReader): array<string, int|string>` and `WordPressTaxonomySource::slugIndex(WxrReader): array<string, int|string>` — build the login/slug lookup tables the resolvers above need, once, from the WXR document.
  - New destination fields (additive, never replacing the raw ones): `WpPostsToArticles` gains `uid` (author), `parent_ref` (page hierarchy, a same-migration self-lookup — requires the parent to appear earlier in WXR document order, same caveat as the existing `WpCommentsToEngagement` `parent_id` self-lookup), and `term_refs` (`list<int|string>`, term memberships). `WpTermsToTaxonomy` gains `parent_ref` (term hierarchy). `WpMediaToEntities` gains `parent_ref` (attached post) — resolves `null` on the first run because media is imported before posts by design, and correctly on a second run once posts exist (the change in resolved value trips `EntityDestination`'s FR-031 change-detection, so the second run is an update, not a no-op).
  - **Docs**: "Resolving references through the id-map" section in `docs/customization.md` with a full wiring example, the `EntityRepository::findBy(['uuid' => ...])` resolver pattern, the field-addition table, and the two ordering caveats above.

### Fixed

- **BREAKING: `WordPressMediaSource::derivePath()` now emits uploads-relative `file_path` values** (G-017). Previously the method preferred the site-root-relative `<wp:attachment_url>` path (e.g. `/wp-content/uploads/2024/06/foo.pdf`) over the uploads-relative `_wp_attached_file` postmeta, so joining the emitted `file_path` to the operator guide's documented `storage/imports/uploads/` layout (`docs/migrating-from-wordpress.md` Option A) produced a path that never resolved — every attachment's local file copy silently no-op'd (381/381 in the originating field report). The connector now:
  - Prefers postmeta `_wp_attached_file` (already uploads-relative per WordPress convention), returned verbatim minus any leading slash — e.g. `2024/06/foo.pdf`.
  - Falls back to `<wp:attachment_url>` only when that postmeta is absent, stripping everything through the URL's `wp-content/uploads/` segment (case-insensitive, percent-decoded, and correct for both year/month and flat upload layouts) instead of returning the full site-root path.
  - Returns `''` when neither source yields a resolvable uploads-relative path (e.g. an off-site/CDN-only `attachment_url` with no `wp-content/uploads/` segment) — consumers must treat `''` as "no file to copy."

  **Migration for existing consumers:** any destination/process plugin that previously compensated for the old shape (e.g. stripping a `wp-content/uploads/` prefix itself before joining to a local uploads root) should remove that workaround — `file_path` now composes directly with `storage/imports/uploads/<file_path>` as documented.
- `Media\MediaCopier` now logs a `warning` (with the resolved absolute `source` and `target` paths) immediately before throwing `WordPressMediaCopyException::sourceNotFound()`, so a missing/unreadable local media source is never silent even if a caller's logger is the only place the failure is observed before the exception is caught/discarded upstream.
- Wired `WordPressBuilderContentDecode` as the first stage of `Migration\WpPostsToArticles`'s default `content` process chain (before shortcode stripping and oEmbed expansion), so both gaps are fixed out of the box for the shipped example migration.

## [0.1.0-alpha.1] - 2026-05-14

First pre-release. Cut alongside the `waaseyaa/migration ^0.1.0-alpha.179` substrate so consumers can `composer require waaseyaa/migrate-source-wordpress:^0.1.0-alpha.1` instead of pinning `dev-main`.

### Added
- **Streaming WXR parser** (`Wxr\WxrReader`) supporting WXR 1.0/1.1/1.2 with skip-with-warning recovery and a strict-mode opt-in.
- **Five source plugins** implementing M-002's `SourcePluginInterface`, each with a `SourceConformanceTestCase`-derived contract test:
  - `Source\WordPressUserSource` (FR-006)
  - `Source\WordPressTaxonomySource` (FR-009)
  - `Source\WordPressMediaSource` (FR-008)
  - `Source\WordPressPostSource` (FR-005) — single source handles posts, pages, and custom post types
  - `Source\WordPressCommentSource` (FR-007)
- **Three process plugins** implementing M-002's `ProcessPluginInterface`:
  - `Process\WordPressShortcodeStrip` (FR-013) with per-tag rewriter callbacks
  - `Process\WordPressOembedExpand` (FR-014, FR-015, FR-016) — opt-in remote resolution via `Process\OembedFetcherInterface`
  - `Process\WordPressMediaRewriteUrl` (FR-013, FR-034, FR-035) with CDN host allowlist
- **Idempotent media copy primitive**:
  - `Media\MediaCopier` (FR-026..FR-029) — local + HTTP source with size + sha256 verification, atomic temp-rename writes
  - `Media\MediaCopyResult` + `Media\MediaCopyOperation` outcome envelope
  - `Media\MediaFetcherInterface` — pluggable HTTP fetcher (no PSR-18 dep)
- **Five default `MigrationDefinition` factory classes** in `Migration\` (FR-018..FR-025) covering the canonical FR-024 chain users → terms → media → posts → comments. `WpPostsToArticles` is shipped as a renameable example.
- **Stable exception types** with string error-code constants (FR-034, FR-045):
  - `Exception\WxrParseException`
  - `Exception\WordPressMediaCopyException`
  - `Exception\WordPressOembedResolutionException`
- **Operator guide** (`docs/migrating-from-wordpress.md`) and **developer guide** (`docs/customization.md`).
- **Test fixtures**: small-site (2 users / 6 terms / 5 posts / 3 attachments / 4 comments), edge-cases/rtl-language.xml (Hebrew + Arabic + bidi marks), edge-cases/large-entries.xml (~120 KB streaming smoke), and per-source 5000-record large fixtures generated on demand for the conformance C5 memory bound.
- **Integration tests** including end-to-end import against the small-site fixture using an in-memory destination, idempotency proof, and known-value round-trip including password discard.

### Compatibility

- PHP: requires `>= 8.5`.
- Substrate: requires `waaseyaa/migration ^0.1.0-alpha.179` and `waaseyaa/foundation ^0.1.0-alpha.179`.

### Notes

- Kernel-boot end-to-end import (real `EntityDestination` + sqlite `MigrationIdMap` + `MigrationRunner`) is intentionally out-of-package — that wiring lives in the consumer's application boot. The package ships factories + plugins; consumers compose them at runtime.
- `user_login` on comment records is left as a string field; resolving the destination account UUID is operator-controlled because `WordPressUserSource` keys `SourceId` by `wp_user_id`, not by login. See [docs/customization.md](docs/customization.md).
